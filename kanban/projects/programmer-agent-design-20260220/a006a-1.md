## 4. Code Patterns and Best Practices

### 4.1 Architecture Patterns

#### Layered Architecture
```
┌─────────────────────────────────────────┐
│         Presentation Layer              │  ← Frontend (React)
├─────────────────────────────────────────┤
│         API Gateway Layer               │  ← Middleware (Auth, CORS, Rate Limit)
├─────────────────────────────────────────┤
│         Router Layer                    │  ← FastAPI Routers
├─────────────────────────────────────────┤
│         Service Layer                   │  ← Business Logic
├─────────────────────────────────────────┤
│         Repository Layer               │  ← Data Access
├─────────────────────────────────────────┤
│         Storage Layer                   │  ← Databases, Cache, Files
└─────────────────────────────────────────┘
```

**Key Principles:**
- **Separation of Concerns:** Each layer has a single responsibility
- **Dependency Direction:** Upper layers depend on lower layers, never reverse
- **Interface-Based Programming:** Use interfaces (abstract classes) for dependencies

#### Design Patterns Applied

**Strategy Pattern:**
```python
# Abstract strategy interface
class Strategy(ABC):

        pass
```

```python
class StrategyFactory:
```

### 4.4 Caching Pattern

**Multi-Level Caching:**
```python
class CacheManager:
    def __init__(self):
        self.l1_cache = {}  # In-memory cache
        self.l2_cache = Redis()  # Redis cache
    
    async def get(self, key: str) -> Optional[Any]:
        # Try L1 first
        if key in self.l1_cache:
            return self.l1_cache[key]
        
        # Try L2
        value = await self.l2_cache.get(key)
        if value:
            self.l1_cache[key] = value  # Promote to L1
            return value
        
        return None
    
    async def set(self, key: str, value: Any, ttl: int = 300):
        # Set both L1 and L2
        self.l1_cache[key] = value
        await self.l2_cache.set(key, value, ex=ttl)
```

### 5.2 Key Data Flows

#### Strategy Execution Data Flow
```
[1] STRATEGY CREATION
POST /strategies → StrategyRegistry.register() → StrategyRepository.save()

[2] BACKTEST SUBMISSION
POST /backtest → Validate → Load Config → Build Service → Execute

[3] DATA LOADING
MarketDataService.get_historical() → MarketRepository.query() → PostgreSQL/Redis

[4] ENGINE SELECTION
Engine Selector → Backtrader/VectorBT based on params

[5] BACKTEST EXECUTION
Backtrader/VectorBT → Signal Generation → Order Execution

[6] RESULT CALCULATION
Metrics Calculator → Sharpe, Drawdown, etc.

[7] RESULT STORAGE
BacktestRepository.save() → DuckDB

[8] CACHE UPDATE
StrategyCache.set()

[9] RESPONSE
JSON response to frontend
```

#### Authentication Data Flow
```
[1] LOGIN REQUEST
POST /auth/login → Validate credentials → Generate JWT

[2] TOKEN VERIFICATION
Auth Middleware → Decode JWT → Get user from SessionRepository

[3] AUTHORIZED REQUEST
Router → Service → Repository → Storage
      ↑
      └─── Depends(get_current_user)
```

### 5.3 Critical Paths

| Path | Description | Bottleneck | Optimization |
|------|-------------|------------|--------------|
| Strategy Backtest | Strategies Router → StrategyRegistry → Backtrader/VectorBT Engine → Backtest Repository → DuckDB | Synchronous backtest execution | Use Celery async tasks |
| Market Data | Market Router → MarketDataService → MarketData Provider → Market Repository → PostgreSQL/Redis | Multiple API calls | Cache results, batch queries |
| User Auth | System Router → Auth Middleware → Session Manager → Session Repository → PostgreSQL/Redis | JWT decoding overhead | Cache user session in Redis |
| Performance Analysis | Analysis Router → PerformanceService → Metrics Calculator → Backtest Repository → DuckDB | Complex calculations | Pre-calculate metrics, use vectorized operations |

### 5.4 Potential Circular Dependencies

```
Strategy → Backtest → Analysis → Strategy
         ↓                    ↓
      Execution           Results
         ↑                    ↑
         └────────────────────┘
```

**Solution:**
- Use event-driven architecture to decouple (Redis Pub/Sub)
- Introduce message queue (Celery)
- Strategy state updates through async events

## 10. Common Pitfalls and Solutions

### 10.1 Common Issues

| Issue | Impact | Solution |
|-------|--------|----------|
| **Incomplete service layer** | High | Complete service layer analysis, identify gaps |
| **Inconsistent caching** | Medium | Implement unified cache interface |
| **Non-uniform error handling** | Medium | Create unified exception classes, global handlers |
| **Missing API version control** | Low | Implement version migration strategy |
| **Synchronous backtesting** | High | Use Celery async tasks |
| **Database connection pool bottlenecks** | Medium | Increase pool size, monitor usage |
| **Low cache hit rates** | Low | Optimize caching strategy, increase TTL |
| **Redis single point of failure** | High | Use Redis Sentinel or Cluster |

### 10.2 Anti-Patterns to Avoid

1. **Fat Controllers:** Business logic should be in Service layer, not Router layer
2. **Direct Database Access in Router:** Always go through Repository layer
3. **Hardcoded Configuration:** Use environment variables or config files
4. **Silent Failures:** Always log errors and return meaningful error messages
5. **N+1 Query Problem:** Use eager loading or batch queries
6. **Leaking Sensitive Data:** Never log passwords, tokens, or personal data
