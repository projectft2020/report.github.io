### 技術決策合理性分析

#### 決策 1：選擇方案 B（智能並發）vs 方案 A（序列）

**決策：** ✅ 合理

**理由：**
- 方案 A（序列）：80 分鐘，零風險，但效率低
- 方案 B（智能並發）：60 分鐘，風險可控，效率提升 25%
- 當前方式：10 分鐘，風險極高

**分析：**
- 對於日常運營，效率提升 25% 是有價值的
- Token 峰值 200k 在安全範圍內（假設 API 配額 >500k/5分鐘）
- 5 分鐘間隔足夠讓 rate limit 計數器重置（大多數 API 使用 1 分鐘或 5 分鐘滑動窗口）

**替代方案：**
- 動態調整：根據 rate limit 反饋自動調整參數（未來可考慮）
- 自適應間隔：首次啟動使用 3 分鐘，如果觸發 rate limit，自動增加到 8 分鐘

#### 決策 2：使用貪心算法分組

**決策：** ✅ 合理

**理由：**
- 問題規模小（<100 任務），O(n log n) 排序 + O(n) 貪心足夠
- 動態規劃雖然能找到最優解，但複雜度高（O(n * limit)），維護成本高
- 貪心算法結果接近最優（對於小規模任務，通常在 95% 以上）

**實現評價：**
```python
# 按預估 Token 排序（小到大）✅
tasks_with_estimate.sort(key=lambda x: x['estimate']['max'])

# 貪心填充 ✅
if task_count < max_concurrent and current_tokens + max_tokens <= effective_limit:
    current_group.append(task)
```

**潛在改進：**
- 如果任務數量 >100，可以考慮使用 bin packing 的近似算法
- 如果任務大小差異很大，可以考慮降序排列（先放大的，小的填充空隙）

#### 決策 3：使用安全係數 0.8

**決策：** ✅ 合理（保守但安全）

**理由：**
- 預估基於歷史數據，實際可能有偏差
- 給 rate limit 預留緩衝
- 可根據實際情況調整（0.7 更保守，0.9 更激進）

**替代方案：**
- 動態安全係數：根據最近 10 次啟動的預估準確性動態調整
  - 如果實際 Token 比預估低 10%，提高到 0.85
  - 如果實際 Token 比預估高 10%，降低到 0.75

#### 決策 4：組間隔 5 分鐘

**決策：** ✅ 合理（假設 API 使用 5 分鐘滑動窗口）

**理由：**
- 大多數 API（OpenAI、Anthropic）使用 1 分鐘或 5 分鐘滑動窗口
- 5 分鐘足夠讓第一組請求完全完成
- 給 API 服務器預留緩衝時間

**潛在風險：**
- 如果 API 使用 1 分鐘窗口，5 分鐘過於保守（可以降到 2-3 分鐘）
- 如果 API 使用 10 分鐘窗口，5 分鐘不夠（需要增加到 8 分鐘）

**建議：**
- 添加配置選項，記錄實際 API 行為
- 根據 rate limit 錯誤調整間隔

#### 決策 5：事件驅動集成

**決策：** ✅ 優秀

**理由：**
- 避免冗餘掃描（只在待辦任務少時掃描）
- 自動觸發，無需 cron
- 與現有系統（Monitor and Refill、Heartbeat）自然集成

**設計評價：**
```
Heartbeat → Monitor and Refill → Scout 掃描 → 智能啟動器
```

**潛在改進：**
- 添加速率限制：即使事件觸發，也限制最小間隔（例如 30 分鐘）
- 添加熔斷機制：如果連續觸發 rate limit，暫停自動啟動

1. **貪心算法在實踐中足夠好**
   - 對於小規模問題，簡單算法往往比複雜算法更好（易理解、易調試、維護成本低）
   - 不要過度設計

2. **預估的準確性是關鍵**
   - 當前基於規則的預估已經不錯（置信度 70%-90%）
   - 未來可以基於歷史數據動態調整，甚至使用機器學習

3. **配置的靈活性比完美更重要**
   - API 行為可能變化，系統需要能快速適應
   - 提供場景預設（保守、正常、激進）比單一參數更有用

4. **事件驅動架構適合這個系統**
   - 自動觸發，減少人工干預
   - 與現有系統集成自然
   - 易於擴展（可以添加更多事件處理器）
