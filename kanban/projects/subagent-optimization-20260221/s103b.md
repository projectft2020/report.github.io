## Executive Summary

基於現有 Sub-Agent 機制的分析,本文設計一套完整的 Sub-Agent 管理系統需求。現有系統具備基本的生命週期管理功能,但在狀態監控、資源管理、錯誤處理與可視化方面存在改進空間。建議系統應實現四層架構:命令層、控制層、監控層與持久化層,以提供完整的子代理程式管理能力。核心優先級為實現狀態追蹤、資源限制與錯誤恢復機制。

## Analysis

### 現有機制分析

基於 `subagents` 工具與 Sub-Agent 上下文結構,現有系統特性如下:

**已有功能:**
- `list`: 列出已生成的 sub-agent 會話
- `kill`: 終止指定的 sub-agent
- `steer`: 向執行中的 sub-agent 發送引導訊息
- 推送式完成通知 (descendant results auto-announced)
- 識別子代會話 (避免輪詢)

**現有上下文結構:**
```
- Label: 任務識別符 (如 20260221-201150-s103b)
- Requester session: 主代理程式會話 (如 agent:main:main)
- Requester channel: 通道識別 (如 telegram)
- Agent session: 子代理程式會話 ID (UUID)
```

**現有限制:**
- 無細粒度狀態追蹤 (只有 running/completed 之分)
- 無資源使用監控 (CPU、記憶體、執行時間)
- 無錯誤日誌聚合與分析
- 無歷史記錄查詢能力
- 無批量操作支援

### 識別的需求模式

基於 Sub-Agent 的典型使用場景,歸納出四個核心需求模式:

1. **生命週期管理** - 建立、執行、監控、終止、清理
2. **狀態可見性** - 即時狀態、進度、資源使用
3. **錯誤管理** - 失敗檢測、重試、日誌收集
4. **歷史追溯** - 任務記錄、效能分析、模式識別

### 風險評估

| 風險 | 可能性 | 影響 | 緩解措施 |
|------|--------|------|----------|
| 資源耗盡 (記憶體洩漏、殭屍進程) | 中 | 高 | 實施資源限制與自動清理機制 |
| 級聯失敗 (sub-agent 失敗影響主流程) | 中 | 高 | 隔離子代執行環境、實施超時機制 |
| 狀態不一致 (多個管理操作衝突) | 低 | 中 | 實施操作鎖定與原子性事務 |
| 可擴展性瓶頸 (大量並發 sub-agents) | 低 | 中 | 設計非同步架構、分片狀態存儲 |

### 機會點

- **自動化負載均衡**: 根據主代理程式與子代理程式的負載動態分配任務
- **智慧調度**: 基於歷史效能數據優化任務分配策略
- **可觀測性增強**: 整合 Prometheus/Grafana 等監控工具
- **策略配置化**: 允許通過配置文件定義不同類型 sub-agent 的管理策略

## Requirements

### FR-1: 基礎生命週期管理

#### FR-1.1: 會話建立
**描述:** 建立新的 sub-agent 會話,包含完整上下文配置。

**需求:**
- 支援自訂 label (任務識別符)
- 支援指定執行超時時間
- 支援傳遞初始化參數
- 自動生成唯一的 agent session ID (UUID)
- 記錄 requester session 與 channel

**優先級:** High
**驗收標準:**
- 建立 sub-agent 時回傳會話 ID
- 上下文資訊正確持久化
- 超時時自動終止會話

#### FR-1.2: 會話終止
**描述:** 正常或強制終止指定的 sub-agent 會話。

**需求:**
- 支援單個會話終止
- 支援批量終止 (按 label、時間範圍、狀態)
- 支援優雅關閉 (允許完成當前步驟)
- 支援強制終止 (立即停止)
- 終止時清理資源與臨時檔案

**優先級:** High
**驗收標準:**
- 終止操作在 5 秒內完成
- 資源清理率達 100%
- 批量終止支援至少 100 個並發會話

#### FR-1.3: 會話清理
**描述:** 清理已完成或失敗的會話記錄。

**需求:**
- 支援定時自動清理 (可配置保留期)
- 支援手動觸發清理
- 清理前備份關鍵日誌 (可選)
- 提供清理統計資訊

**優先級:** Normal
**驗收標準:**
- 預設保留 7 天記錄
- 清理操作不影響執行中的會話

### FR-2: 狀態追蹤與監控

#### FR-2.1: 狀態模型
**描述:** 定義完整的 sub-agent 狀態模型。

**需求:**
```
狀態狀態機:
pending → running → completed
          ↓         ↓
       timeout   failed
          ↓
       killed
```

**狀態屬性:**
- `label`: 任務識別符
- `agent_session_id`: 子代理程式會話 ID
- `requester_session`: 請求者會話
- `status`: 狀態 (pending/running/completed/failed/timeout/killed)
- `created_at`: 建立時間
- `started_at`: 開始執行時間
- `completed_at`: 完成時間
- `progress`: 進度百分比 (0-100)
- `error_code`: 錯誤代碼 (如有)
- `error_message`: 錯誤訊息 (如有)
- `retry_count`: 重試次數

**優先級:** High
**驗收標準:**
- 狀態轉移遵循狀態機規則
- 所有狀態屬性完整記錄

#### FR-2.2: 即時監控
**描述:** 提供即時狀態查詢與監控功能。

**需求:**
- 支援按會話 ID 查詢狀態
- 支援按 label 批量查詢
- 支援按 requester session 查詢
- 支援按狀態過濾
- 提供摘要統計 (總數、各狀態數量)

**優先級:** High
**驗收標準:**
- 查詢回應時間 < 100ms
- 支援至少 1000 個並發查詢

#### FR-2.3: 資源監控
**描述:** 監控 sub-agent 的資源使用情況。

**需求:**
- 監控 CPU 使用率

**優先級:** Normal
**驗收標準:**

#### FR-2.4: 進度追蹤
**描述:** 追蹤 sub-agent 的執行進度。

**需求:**

interface Session {

**存儲層:**
- SQLite (輕量級、嵌入式) 或文件系統 JSON
- 適合中小規模部署 (單機 < 1000 並發)

**日誌系統:**
- Winston (Node.js) 或 Pino
- 支援多傳輸、日誌輪轉

- 內建指標收集
- 可選整合 Prometheus/Grafana

**序列化:**
- JSON (可讀性好)
- MessagePack (效能優化選項)

### 3. 關鍵設計決策

**決策 1: 狀態存儲選擇**
- **選項:** SQLite vs 文件系統
- **建議:** SQLite (事務支援、查詢能力)
- **理由:** 需要複雜查詢與事務保證,SQLite 提供最佳平衡

**決策 2: 推送通知機制**
- **選項:** 內建事件總線 vs 外部 MQ
- **建議:** 內建事件總線 ( EventEmitter 模式)
- **理由:** 減少依賴,滿足當前規模需求

**決策 3: 並發控制**
- **選項:** 全局鎖 vs 標籤級鎖 vs 無鎖
- **建議:** 標籤級細粒度鎖 + 無鎖讀取
- **理由:** 平衡並發性能與一致性需求
