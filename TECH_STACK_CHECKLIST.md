### 步驟 5：評估功能匹配度

**評估標準：**

| 條件 | 權重 | 說明 |
|------|------|------|
| 功能匹配度 | 40% | 現有系統是否滿足功能需求 |
| 技術棧兼容 | 20% | 是否與現有技術棧兼容 |
| 官方支援 | 20% | 是否有官方支援 |
| 已整合 | 20% | 是否已經整合到系統 |

**評分方式：**
```python
def calculate_match_score(system, requirements):
    match_score = 0

    # 功能匹配度 (40%)
    if system.meets_functional_requirements(requirements):
        match_score += 40

    # 技術棧兼容 (20%)
    if system.is_tech_compatible(requirements):
        match_score += 20

    # 官方支援 (20%)
    if system.has_official_support():
        match_score += 20

    # 已整合 (20%)
    if system.is_integrated():
        match_score += 20

    return match_score
```

**輸出：**
```markdown

- 技術棧兼容：[X/20]

- **總分：[X/100]**

```

### 步驟 6：決策

**決策規則：**

| 總分 | 決策 | 說明 |
|------|------|------|
| ≥ 80% | 使用現有系統 | 現有系統足夠，不需新系統 |
| 50-79% | 評估擴展 | 可能需要擴展現有系統 |
| < 50% | 設計新系統 | 現有系統不足，需要新系統 |

## ⚠️ 常見錯誤

### 錯誤 1：跳過檢查步驟

**問題：** 直接開始設計，沒有檢查現有系統

**解決方案：** 強制執行所有檢查步驟

**案例：**
- ❌ 建議使用 ChromaDB，忽略了已有的 QMD
- ✅ 檢查後發現 QMD 已經滿足需求

### 錯誤 2：評估不準確

**問題：** 功能匹配度評估不準確

**解決方案：** 使用標準化的評分表

**案例：**
- ❌ 評估結果主觀，不準確
- ✅ 使用評分表，客觀準確

### 錯誤 3：決策不合理

**問題：** 決策沒有遵循決策規則

**解決方案：** 嚴格遵循決策規則

**案例：**
- ❌ 分數 ≥ 80% 卻設計新系統
- ✅ 分數 ≥ 80% 使用現有系統
