Dashboard 策略設計（測試用）¶

日期: 2026-02-22
任務編號: s101b
狀態: Draft

📋 執行摘要¶

設計一個簡單、易於實現的 Dashboard 策略，用於測試 subagent 協作流程。本策略專注於快速驗證架構設計流程，而非生產級實現。

🎯 設計目標¶

  1. 快速驗證 - 30分鐘內可完成的測試場景
  2. 簡單直觀 - 最小化複雜度，聚焦核心概念
  3. 可執行性 - Designer 和 Developer 能直接按文檔實現
  4. 架構遵循 - 遵循現有系統架構模式

🏗️ 架構原則¶

核心原則¶

  1. 單一職責 - 每個組件專注一個功能
  2. 數據流單向 - 從數據源 → 處理層 → 展示層
  3. 鬆耦合 - 模組間通過接口通信，不直接依賴實現
  4. 可測試性 - 每層都可獨立測試

技術約束¶

  • 避免引入新的服務或框架
  • 使用現有的技術棧
  • 遵循現有的代碼組織結構

📊 Dashboard 結構設計¶

層級架構¶

┌─────────────────────────────────────┐
│         Presentation Layer          │  ← UI 組件
├─────────────────────────────────────┤
│          Business Logic             │  ← 數據處理
├─────────────────────────────────────┤
│           Data Access               │  ← 數據獲取
├─────────────────────────────────────┤
│           Data Sources              │  ← 外部系統
└─────────────────────────────────────┘

組件劃分¶

1. 數據獲取層 (Data Access)¶

職責: 從各種數據源獲取原始數據

DataFetcher (Interface)
├── fetchMetrics() → Map<String, Metric>
├── fetchAlerts() → List<Alert>
└── fetchStatus() → Status

實現: - MockDataFetcher - 測試用，返回靜態模擬數據 - APIDataFetcher - 生產用，調用外部 API（預留）

2. 業務邏輯層 (Business Logic)¶

職責: 處理、聚合、轉換數據

DashboardService
├── aggregateMetrics(metrics) → DashboardData
├── filterAlerts(alerts) → List<Alert>
└── calculateHealthStatus(metrics) → HealthStatus

核心邏輯: - 數據聚合：將多個來源的指標合併 - 閾值告警：檢查指標是否超過預設閾值 - 健康評分：基於指標計算系統健康度

3. 展示層 (Presentation)¶

職責: 將數據渲染為可視化界面

DashboardView (Interface)
├── renderOverview() → HTML/Component
├── renderCharts() → ChartComponents
└── renderAlerts() → AlertList

實現: - SimpleHTMLDashboard - 基礎 HTML 表格展示 - ReactDashboard - React 組件（預留擴展）

🔄 數據流設計¶

主流程¶

1. User Request
   ↓
2. Controller → dashboardService.generateDashboard()
   ↓
3. DashboardService → dataFetcher.fetchMetrics()
   ↓
4. DataFetcher → External Data Source
   ↓
5. DashboardService → aggregate(), filter(), calculate()
   ↓
6. DashboardService → DashboardData
   ↓
7. Controller → dashboardView.render(DashboardData)
   ↓
8. Response → User

錯誤處理策略¶

Data Source Error
├── Retry (3次，指數退避)
├── Fallback → 返回緩存數據
└── Degraded Mode → 顯示「數據不可用」標識

📐 數據結構定義¶

核心數據類型¶

// 指標數據
interface Metric {
  id: string;
  name: string;
  value: number;
  unit: string;
  timestamp: Date;
  threshold?: number;
}

// 告警信息
interface Alert {
  id: string;
  level: 'info' | 'warning' | 'error' | 'critical';
  message: string;
  timestamp: Date;
  source: string;
}

// 系統狀態
interface Status {
  healthy: boolean;
  uptime: number;
  lastUpdate: Date;
}

// Dashboard 數據聚合
interface DashboardData {
  metrics: Map<string, Metric>;
  alerts: List<Alert>;
  status: Status;
  healthScore: number; // 0-100
  generatedAt: Date;
}

測試場景設計¶

場景 A: 正常顯示¶

  • 所有數據源正常
  • 顯示完整 Dashboard
  • 驗證數據流完整性

場景 B: 告警觸發¶

  • CPU > 80% 觸發警告
  • 顯示告警列表
  • 驗證閾值邏輯

場景 C: 數據源故障¶

  • API 超時
  • 驗證 fallback 機制
  • 顯示降級模式
