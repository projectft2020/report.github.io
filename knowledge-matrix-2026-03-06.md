### 背壓機制模式（Backpressure Mechanism）

#### 核心理念
```
容量規劃原則：
- 不在 100% 容量運作
- 80% 是安全線
- 50% 是舒適線
```

#### 可複用模式
```
┌─────────────────────────────────────────────────────────┐
│                   背壓機制模式                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 監測階段                                           │
│     ├── 監測指標：並發數、啟動頻率、健康度           │
│     ├── 卡住任務數量                                   │
│     └── 完成率                                         │
│                                                         │
│  2. 計算健康度                                         │
│     ├── health_score = max(0, 1 - stuck_ratio)       │
│     ├── stuck_ratio = stuck_tasks / total_tasks       │
│     └── 目標：health_score >= 0.67                     │
│                                                         │
│  3. 動態調整                                           │
│     ├── health_score >= 0.8 → 啟動頻率 65 秒         │
│     ├── 0.67 <= health_score < 0.8 → 啟動頻率 300 秒  │
│     ├── health_score < 0.67 → 啟動頻率 900 秒         │
│     └── 並發上限：max_concurrent = max_concurrent * health_score │
│                                                         │
│  4. 保護機制                                           │
│     ├── 最小並發：1 個                                 │
│     ├── 最大並發：10 個                                │
│     ├── 啟動頻率限制：60-1200 秒                       │
│     └── 卡住任務回滾                                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 實證數據

### 三層超時機制（Three-Tier Timeout）

#### 核心理念
```
超時分層原則：
- 快速失敗（30 秒）→ 中等警告（30 分鐘）→ 自動回滾（45 分鐘）→ 執行超時（90 分鐘）
```

#### 超時配置
```
┌─────────────────────────────────────────────────────────┐
│                 三層超時機制配置                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  第一層：啟動超時（30 秒）                              │
│  ├─ 目標：快速檢測啟動失敗                             │
│  ├─ 行動：記錄警告，發送通知                           │
│  └─ 頻率：每 30 秒檢查一次                             │
│                                                         │
│  第二層：Spawning 超時（30 分鐘）                     │
│  ├─ 目標：檢測卡在 spawning 狀態的任務                 │
│  ├─ 行動：記錄警告，標記為疑似卡住                     │
│  └─ 頻率：每 5 分鐘檢查一次                           │
│                                                         │
│  第三層：自動回滾（45 分鐘）                           │
│  ├─ 目標：自動清理卡住的任務                           │
│  ├─ 行動：標記為 'ready'，清理 .status 文件            │
│  └─ 頻率：每 10 分鐘檢查一次                          │
│                                                         │
│  第四層：執行超時（90 分鐘）                           │
│  ├─ 目標：檢測長時間執行的任務                         │
│  ├─ 行動：標記為失敗，清理資源                         │
│  └─ 頻率：每 15 分鐘檢查一次                          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 實證數據
```
回滾時間對比：
- 優化前：120 分鐘（手動檢測 + 手動回滾）
- 優化後：45 分鐘（自動檢測 + 自動回滾）
- 改善：62.5%

卡住任務數量對比：
- 優化前：6 個（平均）
- 優化後：0-2 個（自動回滾）
- 改善：100%（6 → 0，穩定狀態）
```

#### 關鍵洞察
1. **自動化回滾 > 手動回滾** - 效率提升 62.5%
2. **超時機制需要分層** - 不是一刀切
3. **系統自我修復能力** - 從被動響應到主動防禦

```

   ├── 引入「大小元組」新度量
   ├── 引入「維度向量」新度量
   ├── 細緻地表徵 Transformer 結構
   └── 推導 Lipschitz 常數和記憶容量的上界
```

#### 意義
```
1. 理論基礎
   ├── 為大語言模型的成功提供理論依據
   ├── 解釋 Transformer 為什麼有效
   └── 填補了理論空白

2. 架構設計
   ├── 對 Transformer 架構設計具有指導價值
   ├── 優化層數、注意力頭數、嵌入維度

   └── 提高訓練效率

4. 局限性
   ├── 存在對數因子 log^2 m
   ├── 某些情況下的常數因子可能較大
   └── 需要進一步優化
```

#### 關鍵洞察
1. **Transformer 逼近能力首次被嚴格證明** - 理論基礎

```
背壓機制：

- 並發上限：max_concurrent = max_concurrent * health_score
- 啟動頻率：65 秒（健康）、300 秒（一般）、900 秒（不健康）

三層超時：
- 啟動超時：30 秒（快速失敗）
- Spawning 超時：30 分鐘（中等警告）

- 執行超時：90 分鐘（執行失敗）

- 先事實，後判斷
- 先客觀，後主觀
```
