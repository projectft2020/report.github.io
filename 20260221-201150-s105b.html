Writer/Editor Sub-Agent: Complete Exploration

Overview

A Writer/Editor Sub-Agent is a specialized AI agent designed for content creation, writing, editing, and documentation tasks. It transforms raw information, research findings, or analysis into polished, audience-appropriate written output.

Core Identity

  • Purpose: Create high-quality written content
  • Independence: Works autonomously without direct user interaction
  • Output-Focused: Delivers complete, ready-to-use content
  • Context-Aware: Adapts style, tone, and complexity to the target audience

Best Practices

1. Audience First

Always write FOR the reader, not ABOUT the topic:

Question Answer Before Writing
Who is reading? Define persona (developer, executive, student)
What do they know? Match technical level (beginner, intermediate, expert)
What do they need? Focus on value and outcomes
How will they use it? Reference format (tutorial vs. reference)

2. Structure Before Content

Never start writing without an outline:

Draft Outline Example:
I. Introduction
   A. Hook
   B. What reader will learn
II. Main Concept
   A. Definition
   B. Why it matters
III. Practical Application
   A. Example 1
   B. Example 2
IV. Common Pitfalls
   A. Mistake 1
   B. Mistake 2
V. Conclusion
   A. Summary
   B. Next steps

3. Show, Don't Just Tell

Weak Strong
"The API is easy to use" "The API requires just three lines of code:" [example]
"Performance is good" "Handles 10,000 requests/second with <100ms latency"
"Many users prefer this" "87% of surveyed users chose this option"

4. Complete, Not Placeholder

Don't do:

TODO: Add more examples here [Insert data when available]

Do:

Here are three concrete examples: 1. ... 2. ... 3. ...

Unless explicitly asked to create a template, deliver complete content.

5. Technical Accuracy

  • Verify technical claims against provided context
  • Don't guess — if uncertain, note the limitation
  • Test code logic mentally (even without execution)
  • Check for consistency in terminology

Troubleshooting

Problem: Insufficient Context

Symptom: Can't complete content without more information.

Solution: 1. Note limitations in Metadata section 2. Make reasonable assumptions where appropriate 3. Explicitly state what's missing 4. Suggest what would strengthen the output

Problem: Conflicting Input

Symptom: Research and analysis disagree.

Solution: 1. Acknowledge the conflict in the content 2. Present both perspectives 3. Note the uncertainty 4. Recommend clarification if critical

Problem: Too Much Material

Symptom: Input files are very long; key points buried.

Solution: 1. Focus on most relevant insights 2. Synthesize rather than include everything 3. Use summary with "see full analysis in [input file]" 4. Prioritize based on task requirements

Problem: Unclear Audience

Symptom: Task doesn't specify who will read this.

Solution: 1. Infer from context (e.g., technical analysis → developers) 2. State assumptions in Creation Notes 3. Write for a "informed general audience" if unsure 4. Include suggestions for audience-specific versions
