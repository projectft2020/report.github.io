# 第 3 階段使用指南：自動分解建議

## 概述

第 3 階段提供自動化的任務分解建議功能，根據複雜度自動判斷任務是否需要分解，並生成具體的分解方案。

## 何時需要分解？

### 自動判斷規則

任務在以下情況會被建議分解：

| 條件 | 說明 |
|------|------|
| 複雜度等級 4 | 極複雜任務，必須分解 |
| 複雜度等級 3 + 輸入文件 ≥ 3 | 文件太多，建議分組 |
| 複雜度等級 3 + analyst 代理 | 分析任務較複雜 |
| 複雜度分數 > 7 | 超過複雜度閾值 |

### 不需要分解的情況

| 複雜度等級 | 典型特徵 | 建議 |
|-----------|----------|------|
| 等級 1 | 0-1 個輸入，簡單任務 | 直接執行 |
| 等級 2 | 1-2 個輸入，中等任務 | 直接執行 |

---

## 分解策略

### 策略 1：按輸入文件分解 (by_input)

**適用情況**：輸入文件 ≥ 3 個

**方法**：
- 將輸入文件分組（每組 1-2 個）
- 每組成為一個獨立子任務
- 子任務按順序執行

**範例**：
```
原始任務：4 個輸入文件 [h003c, h002, h003a, h003b]

分解為：
  h004-a: 處理 [h003c, h002]
  h004-b: 處理 [h003a, h003b]

依賴：h004-b 依賴 h004-a
```

### 策略 2：按階段分解 (by_phase)

**適用情況**：任務包含分析、驗證、實證、測試等階段

**方法**：
- 階段 1：準備與分析
- 階段 2：實作與生成
- 階段 3：驗證與測試（如需要）

**範例**：
```
原始任務：回測驗證

分解為：
  h004-a: 階段 1：準備與分析
  h004-b: 階段 2：實作與生成（依賴 h004-a）
  h004-c: 階段 3：驗證與測試（依賴 h004-b）
```

### 策略 3：按組件分解 (by_component)

**適用情況**：任務包含集成、整合、系統、框架等關鍵詞

**方法**：
- 組件 1：架構設計
- 組件 2：核心實作
- 組件 3：集成與測試

**範例**：
```
原始任務：系統集成

分解為：
  qe002-a: 組件 1：架構設計
  qe002-b: 組件 2：核心實作（依賴 qe002-a）
  qe002-c: 組件 3：集成與測試（依賴 qe002-a, qe002-b）
```

## 分解效果預估

### 時間節省計算

```
原始時間 = (預估最小 + 預估最大) / 2
子任務時間 = 13 分鐘（等級 2 的平均時間）
總子任務時間 = 子任務時間 × 子任務數量
並行後時間 = 總子任務時間 × 0.7（假設 30% 可並行）

時間節省 = ((原始時間 - 並行後時間) / 原始時間) × 100%
```

### 範例計算

**h004 任務**：
- 原始預估：25 分鐘（等級 3 中點）
- 分解為：2 個子任務
- 並行後：13 × 2 × 0.7 = 18.2 分鐘
- 時間節省：27.2%

### 成功率提升

| 情況 | 不分解 | 分解後 |
|------|--------|--------|
| 超時風險 | 高（60% 成功率）| 低（95% 成功率）|
| 錯誤隔離 | 困難 | 容易（每個子任務獨立）|
| 重試成本 | 高（整個任務）| 低（只需重試失敗的子任務）|

## 使用 Python API

### TaskDecomposer 類

```python
from task_decomposer import TaskDecomposer

decomposer = TaskDecomposer()

# 分析任務
task = {
    'id': 'h004',
    'title': 'adaptive-hedge 回測驗證',
    'agent': 'automation',
    'input_paths': ['h003c', 'h002', 'h003a', 'h003b'],
    'notes': 'adaptive-hedge 回測驗證：歷史崩盤事件實證測試'
}

suggestion = decomposer.analyze_decomposition(task)

# 檢查是否需要分解
if suggestion.should_decompose:
    print(f"需要分解：{suggestion.reason}")
    print(f"預估節省：{suggestion.estimated_time_reduction}%")

    # 遍歷子任務
    for subtask in suggestion.subtasks:
        print(f"  {subtask.id}: {subtask.title}")
else:
    print(f"不需要分解：{suggestion.reason}")
```

## 故障排除

### 問題 1：所有任務都被建議分解

**原因**：複雜度評估規則太嚴格

**解決**：
- 調整 `_should_decompose` 方法的閾值
- 提高複雜度分數閾值（從 7 到 8）

### 問題 2：分解後任務仍然失敗

**原因**：
- 子任務仍然太複雜
- 模型選擇錯誤

**解決**：
- 檢查子任務的複雜度等級
- 使用 GLM-4.7（如果等級 3）
- 進一步分解

### 問題 3：分解後時間更長

**原因**：
- 子任務無法並行
- 依賴關係太多

**解決**：
- 考慮不同的分解策略
- 減少依賴關係
- 使用並行執行

## 最佳實踐

### 1. 創建任務前先檢查

```bash
# 1. 先分析
python3 task_manager.py analyze p006

# 2. 如果建議分解，先看分解方案
python3 task_manager.py decompose p006

# 3. 決定是否分解或直接執行
```

### 2. 選擇合適的分解策略

- 輸入多 → 按輸入文件
- 有階段性 → 按階段
- 系統集成 → 按組件

### 3. 確保子任務複雜度 ≤ 等級 2

分解後的每個子任務應該：
- 輸入文件 ≤ 2 個
- 複雜度分數 ≤ 4
- 可獨立執行

### 4. 設置合理的依賴

- 避免循環依賴
- 最大化並行度
- 清晰的數據流
