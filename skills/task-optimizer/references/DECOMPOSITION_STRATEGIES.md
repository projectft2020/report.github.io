# 任务分解策略

## 何时需要分解

### 判断规则
需要分解 IF 满足以下任一条件：
- ✅ 复杂度等级 = 4
- ✅ 复杂度等级 = 3 且 输入文件 ≥ 3
- ✅ 复杂度等级 = 3 且 agent = analyst
- ✅ 复杂度分数 > 7

### 不需要分解的情况
- ❌ 复杂度等级 1-2（简单/中等）
- ❌ 复杂度等级 3 且 输入文件 ≤ 2
- ❌ 任务类型 = automation 或 research（通常不需要）

## 三种分解策略

### 策略 1：按输入文件分解（最常用）
**适用场景：**
- 输入文件 ≥ 3 个
- 任务需要处理多个独立文件

**方法：**
- 每组 1-2 个文件
- 创建 2-3 个子任务
- 最后汇总所有结果

**示例：**
```
原始任务：分析 4 个策略文件
输入：strategy1.md, strategy2.md, strategy3.md, strategy4.md

分解：
- 子任务 A：分析 strategy1.md + strategy2.md
- 子任务 B：分析 strategy3.md + strategy4.md
- 子任务 C：汇总 A 和 B 的结果
```

**时间节省：** 20-30%

### 策略 2：按阶段分解
**适用场景：**
- 任务有明确的阶段性流程
- 需要 验证/测试 阶段

**方法：**
- 识别主要阶段（分析 → 实作 → 验证）
- 每个阶段一个子任务
- 按顺序执行

**示例：**
```
原始任务：开发交易系统
阶段：
- 子任务 A：系统设计
- 子任务 B：代码实现
- 子任务 C：回测验证
- 子任务 D：性能优化
```

**时间节省：** 15-25%

### 策略 3：按组件分解
**适用场景：**
- 系统集成类任务
- 有多个独立组件

**方法：**
- 识别主要组件（前端、后端、数据库）
- 每个组件一个子任务
- 最后集成

**示例：**
```
原始任务：Dashboard 集成
组件：
- 子任务 A：前端集成
- 子任务 B：后端集成
- 子任务 C：数据库迁移
- 子任务 D：端到端测试
```

**时间节省：** 25-35%

## 分解后的模型选择

### 规则
```
子任务复杂度等级 1-2 → GLM-4.5
子任务复杂度等级 3-4 → GLM-4.7
```

### 示例
原始任务：等级 4（4 个输入文件）
分解后：
- 子任务 A：等级 2（2 个输入）→ GLM-4.5
- 子任务 B：等级 2（2 个输入）→ GLM-4.5
- 汇总任务：等级 2（2 个输出）→ GLM-4.5

## 最佳实践

1. **优先使用按输入文件分解**
   - 最简单、最直接
   - 子任务独立性高
   - 容易并行执行

2. **避免过度分解**
   - 每个子任务至少 5 分钟工作量
   - 分解不应增加管理成本

3. **考虑汇总任务**
   - 大部分分解需要最后的汇总
   - 汇总任务应该明确

4. **记录分解决策**
   - 在 tasks.json 中标记 `depends_on`
   - 保持依赖关系清晰
