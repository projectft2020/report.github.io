## 核心學習（續）

### 本地 LLM 推理的價值
- **成本效益：** Embedding 模型本地化可立即節省成本
- **延遲優勢：** 本地推理延遲 < 100ms，遠低於雲端 200-500ms
- **離線能力：** 飛機上、網絡不佳時仍可工作
- **整合漸進：** 不必一次性替換所有雲端服務，可以漸進式整合

### 分層 KV 緩存技術
- **熱層（RAM）：** 頻繁訪問的 KV 塊保持記憶體中，快速訪問
- **冷層（SSD）：** 熱層滿時，將冷塊卸載到 SSD（safetensors 格式）
- **前綴共享：** 多個請求共享相同前綴，避免重複計算
- **Copy-on-Write：** 分支請求共享未修改部分，節省記憶體

### 漸進式驗證策略
- **階段 1（低風險）：** Embedding 試用（模型小，即時收益）
- **階段 2（中等風險）：** LLM 試用（需評估質量）
- **階段 3（整合）：** 與現有工具整合（需要配置時間）
- **隨時退出：** 如不合適，隨時卸載（無沉沒成本）

### oMLX vs 雲端 API 權衡

| 維度 | oMLX（本地） | 雲端 API |
|------|-------------|----------|
| **成本** | 免費（硬件已存在） | 按量計費 |
| **延遲** | < 100ms | 200-500ms |
| **可用性** | 離線可用 | 需網絡 |
| **質量** | 量化模型（輕微下降） | 全精度模型 |
| **維護** | 需手動管理 | 自動更新 |
| **可擴展性** | 受硬件限制 | 幾乎無限 |

#### 根本原因：不是「佔用太多」，而是「單次分配超限」

**錯誤訊息：** `kIOGPUCommandBufferCallbackErrorOutOfMemory`

**真實原因：**
- 當 QMD 嘗試向量化大文件（110K）時
- 文件被分成很多 chunks（900 tokens/chunk，約 150 個）
- QMD 嘗試一次性向量化所有 chunks
- Apple M4 Metal GPU 雖然有 10.7GB，但**單次操作有分配限制**
- 就像冰箱很大，但一次塞入一整箱大西瓜，放不進單層

**類比：**
- 你的冰箱：10.7GB（夠大）
- 嘗試一次塞入：一整箱大西瓜（110K 文件的向量）
- 箱子太大，放不進冰箱的單層
- 即使冰箱還是空的，也會失敗

- **不是「空間不足」，而是「單次分配超限」**
