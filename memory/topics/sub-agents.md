## 🎯 模型使用策略

### 主模型：zai/glm-4.7-flash
- **默認模型**：所有會話
- **特點：** 快速、高效、成本低
- **適用：** 快速響應、簡單任務、背景執行
- **並發限制：** 1 個同時運行

### 備用模型：zai/glm-4.5
- **觸發條件：** 主模型限流時自動切換
- **特點：** 穩定、可靠
- **適用：** Research、Automation、Creative 子代理
- **並發限制：** 無（可同時運行多個）

### 高品質模型：zai/glm-4.7
- **使用時機：** 深度思考、複雜分析、架構設計
- **特點：** 高品質輸出、複雜推理
- **適用：** Analyst、Architect、Mentor
- **並發限制：**
  - GLM-4.7-Flash = 1 個同時運行
  - Mentor = 1 個同時運行
  - Architect = 1 個同時運行
  - Developer = 2 個同時運行

## 🔧 調試和診斷

### 常見失敗原因
1. **超時失敗**（runtime > 估計時間）
   - 原因：任務複雜度超出預期
   - 解決：任務分解、增加時間估算

2. **假失敗**（輸出文件存在但標記 failed）
   - 原因：API 超時或臨時錯誤
   - 解決：檢查輸出文件，更正狀態

3. **子代理失敗**（automation 代理 0% 可靠）
   - 原因：agent 實現問題
   - 解決：切換到可靠代理（research/analyst）

## 📈 最佳實踐

### ✅ 最佳實踐
1. **Research Agent** - 網絡搜索和事實查詢
2. **Analyst Agent** - 數據分析和邏輯推理
3. **Task Decomposition** - 複雜任務拆分為小任務
4. **Parallel Execution** - 一次啟動 5 個任務（最大並發）
5. **Model Selection** - 根據任務複雜度選擇適合的模型

### ❌ 避免實踐
1. **Automation Agent** - 複雜任務容易超時失敗
2. **單模型執行** - 避免在 GLM-4.7-Flash 上運行多個高複雜任務
3. **忽略輸出驗證** - 不要只看任務狀態，要檢查輸出文件
4. **過度分解** - 避免將簡單任務過度拆分
