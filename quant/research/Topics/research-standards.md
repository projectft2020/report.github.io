## 📚 研究報告標準（m001 範例）

### 11 個標準部分

#### 1. Executive Summary（執行摘要）
- 簡要概述研究目標和核心發現
- 適合快速了解研究內容

#### 2. 公式定義（完整數學表達式）
- 完整的數學公式，包含變數定義
- 標準的數學符號和表達式

#### 3. 數學推導（從基本概念到最終公式）
- 從第一性原理開始
- 逐步推導過程
- 詳細的中間步驟

#### 4. 特徵分析（優點、缺點、應用場景）
- 數學性質
- 金融解釋
- 實際應用場景
- 計算效率

#### 5. 特殊情況（閉合解、極限情況）
- 正態分佈的閉合解
- t-分佈的閉合解
- 極限情況（如 n→∞）

#### 6. 學術來源（主要論文 + 參考文獻）
- 主要論文（完整引用）
- 參考文獻（相關研究）
- 期刊和會議

#### 7. 指標對比表（多個指標一覽）
- 公式對比
- 特徵對比
- 優點/缺點對比
- 適用場景對比

#### 8. 實施建議（指標選擇、計算流程、應用場景）
- 指標選擇指南
- 計算流程
- 分階段部署建議
- 應用場景推薦

#### 9. 與傳統方法對比
- 與 Sharpe Ratio 對比
- 與傳統風險度量對比
- 優缺點分析

#### 10. 結論與參考文獻
- 總結論
- 未來研究方向
- 完整參考文獻

#### 11. 完整 Python 代碼框架
- 可直接運行的代碼
- 包含類和函數
- 完整的錯誤處理
- 使用示例

## 📏 代碼規範

### Python 代碼標準
- **類命名：** PascalCase（如 DailyHedgeRatioIndicator）
- **函數命名：** snake_case（如 calculate_hedge_ratio）
- **導入：** `import numpy as np`, `import pandas as pd`
- **類型提示：** 完整的類型提示
- **文檔字串：** 完整的函數和類型文檔字串

### 測試標準
- **單元測試：** 測試每個函數和類
- **集成測試：** 測試完整流程
- **邊界測試：** 測試極端情況和錯誤輸入

### 錯誤處理
- **try-except 模式：** 所有可能的錯誤都捕獲
- **日誌記錄：** 使用 logger 記錄錯誤
- **優雅失敗：** return safe defaults，不拋出異常

## 🔍 數據驗證重要性（2026-02-28 學習）

### 核心原則：數據驗證是研究的生命線

**關鍵洞察：** 不能盲目相信回測數據，需要驗證邏輯一致性。

### 數據驗證檢查清單

#### 1. 邏輯一致性檢查
- 勝率 100% 是否合理？
- 最大虧損與勝率是否矛盾？
- 獲利因子計算是否正確？
- 平均報酬計算是否一致？

#### 2. 概率分析檢查
- 勝率 100% 的概率是多少？
- 統計上是否可能？
- 是否需要數學驗證？

#### 3. 數據洩漏檢查
- 回測時是否使用了未來的數據？
- 例如：在第 0 天就知道第 7 天的虧損
- 是否有 look-ahead bias？

#### 4. 計算方式一致性檢查
- "總報酬" 的定義是否清楚？
- 平均報酬的計算方式是否一致？
- 是否有隱藏的複利計算？

### 驗證案例：Supertrend 策略對比數據

#### 問題 1：延遲+動態策略勝率 100% - 不可能
- 214 筆交易全部獲利
- 概率：0.0000000000 ≈ 1 / 2.63 × 10⁴³
- **結論：幾乎不可能，數據造假或過擬合**

#### 問題 2：最大虧損 1.37% 與勝率 100% 矛盾
- 如果勝率 100%，所有交易都盈利
- 為什麼還有最大虧損 1.37%？
- **結論：邏輯矛盾，計算混亂**

#### 問題 3：獲利因子 0.00 - 計算錯誤
- 獲利因子 = 總盈利 / 總虧損
- 如果勝率 100%，總虧損 = 0
- 獲利因子應為無窮大
- **結論：計算錯誤，除以 0 異常處理錯誤**

#### 問題 4：原始策略平均報酬計算錯誤
- 總報酬：234.07%
- 交易筆數：466
- 平均報酬計算：234.07 / 466 = 0.50%
- 聲稱平均報酬：11.17%
- 差異：10.67%
- **結論：計算方式不一致，可能是複利計算但未說明**

#### 問題 5：延遲進場交易筆數未過濾
- 原始策略：466 筆
- 延遲進場：466 筆（相同）
- 應該有部分交易被過濾（第 8 天時已出場）
- **結論：邏輯不合理**

### 可能的原因

#### 1. 數據洩漏（未來函數）🔴
- 回測時使用了未來的數據
- 例如：在第 0 天就知道第 7 天的虧損
- 這會導致過擬合和不真實的績效

#### 2. 計算邏輯錯誤 🟡
- "總報酬" 的定義不清楚
- 平均報酬的計算方式不一致
- 獲利因子計算錯誤

#### 3. 數據偽造 🔴
- 勝率 100% 不可能
- 為了讓延遲+動態策略看起來優秀而偽造數據

### 重要教訓

#### 1. 數據驗證很重要
- 不能盲目相信回測數據
- 需要驗證邏輯一致性
- 勝率 100% 等不合理數據要警惕

#### 2. 假設驗證很重要
- 我之前基於錯誤假設（固定天數出場）給出錯誤建議
- 正確的假設是：訊號驅動出場
- 需要清楚定義回測框架

#### 3. 數據洩漏是回測的大敵
- 未來函數會導致過擬合
- 需要嚴格檢查是否使用了未來的數據
- 真實的績效不會過於完美

#### 4. 統計概率 > 直覺判斷
- 勝率 100% 在統計上不可能
- 需要基於概率分析評估策略可信度
- 數據驗證腳本很有用

### 推薦的正確策略（基於 462 筆實際數據）

```
第 0 天：收到 Supertrend 信號 → 立即進場
第 7 天：評估虧損
  - 虧損 > 3% → 出場（假突破）
  - 虧損 1-3% → 減倉 50%，觀察
  - 虧損 < 1% → 持倉
第 11 天：確認趨勢
  - 盈利 > 0 → 加倉 50%
  - 仍虧損 < 1% → 持倉
  - 虧損 > 2% → 出場
出場：收到反向 Supertrend 信號
```

**為什麼選擇動態風險管理？**
1. 第 7 天評估虧損，過濾假突破
2. 第 11 天確認趨勢，加倉盈利交易
3. 訊號驅動出場，捕捉長期趨勢
4. 動態調整倉位，優化風險收益比

## 📊 樣本內/樣本外測試

### 目的
- **驗證：** 模型的泛化能力
- **避免：** 過度擬合（overfitting）

### 方法
1. **樣本內測試** - 使用部分數據訓練，部分數據測試
2. **樣本外測試** - 使用歷史數據訓練，未來數據測試
3. **Walk-Forward 分析** - 時間序列交叉驗證

### 指標
- **泛化能力：** 訓練集 vs 測試集的表現差異
- **穩定性：** 不同時間窗口的表現穩定性
- **魯棒性：** 市場變化下的表現
